## 原因：比较前会发生“整型提升”

在 Keil 里，`uint` 一般是：

```c
typedef unsigned int uint;   // 通常 32 位无符号
```

`short` 是 **16 位有符号**（-32768 ~ 32767）。

比较例如 `if (u > s)` 时，编译器大致会做：

1. `short` 先提升为 `int`
2. 再与 `unsigned int` 比较时，**有符号的一方会转成无符号**
3. 若 `short` 为 **负数**，转成 `uint` 后会变成一个 **很大的正数**（例如 -1 → 4294967295）

### 典型错误例子

```c
uint  u = 5;
short s = -1;

if (u > s)   // 你以为：5 > -1 → 真
             // 实际：5 > 4294967295 → 假
{
    // 进不来
}
```

再例如：

```c
uint  u = 100;
short s = 50;

if (u < s)   // 100 < 50？你以为假
             // 若 s 为负，s 会变成很大的无符号数，结果可能为真
```

所以：**只要 `short` 可能为负，和 `uint` 直接比较就不可靠。**

## 什么时候“看起来还对”？

仅当 `short` **确定永远 ≥ 0**（例如只做 0~32767 的计数、且上游不会赋负值）时，比较有时碰巧正确——但这是 **靠运气**，维护时一旦有人赋了 `-1` 或传感器给负值，就会 silently 出错。嵌入式里不建议这么写。

## 正确写法（按场景选）

**1. 比较前统一类型（推荐）**

```c
// short 可能为负：先转成有符号宽类型再比
if ((int32_t)u > (int32_t)s) { ... }

// 或保证 s 非负再比
if (s >= 0 && u > (uint)s) { ... }
```

**2. 从设计上避免混用**

- 表示“不可能为负”的量：用 `uint` / `unsigned short`
- 表示“可能为负”的量：用 `int` / `int16_t`，**不要**和 `uint` 直接比

**3. 赋值也要注意**

```c
uint u;
short s = -1;
u = s;        // u 会变成很大的数，不是 -1
```

你们手册里也强调：**嵌入式用简单、类型一致的变量，方便 Keil 搜索和排查。** `uint` 和 `short` 混比，正是这类“表面能编译、运行却错”的坑。

## 3. 为什么容易误以为“该用 int 不该用 short”？

常见两种写法混在一起：

| 写法 | -1 时表现 |
|------|-----------|
| `if (val == -1)` | `int` / `short` 都对 |
| `if ((uint)val > xxx)` 或 `val` 与 `uint` 比大小 | `int` / `short` 都可能错 |

## CAN 上常见情况

CAN 一帧里 2 字节拼成 16 位，例如：

```text
