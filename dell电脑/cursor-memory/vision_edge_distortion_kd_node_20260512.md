```text
第三嫌疑：边缘成像和角点定位质量
第四嫌疑：当前 5 参数模型不足，需要 rational model 或残差补偿
```

## 5. 等效补偿模型

由于没有原始四角像素 `P_c1~P_c4`，无法真正重新求 OpenCV 的 K/D。

因此做了一个工程上的等效补偿模型：

```text
当前 K/D -> 程序输出 center_x/center_y/tx/ty/tz
然后根据 center_x/center_y 对 tx/ty/tz 做后处理补偿
再用 corrected T_world_cam 转世界坐标
```

补偿模型：

```text
xn = (center_x - cx) / fx
yn = (center_y - cy) / fy
r2 = xn*xn + yn*yn

X = tx * 1000
Y = ty * 1000
Z = tz * 1000

Xc = X + (ax*xn + bx*xn*r2) * Z
Yc = Y + (ay*yn + by*yn*r2) * Z
Zc = Z * (1 + az*r2 + bz*r2*r2)
```

补偿参数：

```cpp
double ax =  0.000856;
double bx = -0.140546;

double ay = -2.006687;
double by =  0.087909;

double az = -0.856645;
double bz =  2.152915;
```

补偿后 21 点结果：

```text
平均误差 = 11.05mm
RMS误差  = 12.99mm
最大误差 = 30.44mm
```

说明：

```text
1. 误差具有明显图像位置规律。
2. 当前边缘误差可以被 center_x/center_y 相关的残差模型显著补偿。
3. 这不是严格 K/D，只是当前相机+当前K/D输出后的工程补偿层。
```

## 8. 补偿逻辑加入程序的位置

补偿不加在亚像素之前，也不加在 `estimate_tag_pose` 之前。

正确链路：

```text
图像
-> AprilTag 检测
-> 角点亚像素
-> 当前 K/D + TagSize 估计 pose
-> 得到 tx/ty/tz
-> 用 center_x/center_y 做等效补偿
-> 得到 tx_comp/ty_comp/tz_comp
-> 后续世界坐标转换使用补偿值
```

应加在：

```cpp
estimate_tag_pose(&info, &pose);

result.tx = pose.t->data[0];
result.ty = pose.t->data[1];
result.tz = pose.t->data[2];
```

之后。

建议在 `AprilTagDetection` 结构体新增：

```cpp
double tx_comp = 0.0;
double ty_comp = 0.0;
double tz_comp = 0.0;
double distance_comp = 0.0;
```

不要覆盖原始 `tx/ty/tz`，便于调试对比。

## 9. C++ 补偿函数

```cpp
struct CompensatedCamPoint
{
    double x_mm;
    double y_mm;
    double z_mm;
};

CompensatedCamPoint CompensateTagCamPoint(
    double center_x,
    double center_y,
    double tx_m,
    double ty_m,
    double tz_m)
{
    const double fx = 2856.4544591714798;
    const double fy = 2886.8241242873419;
    const double cx = 1055.0632884691124;
    const double cy = 552.05322013539921;

    const double ax =  0.000856;
    const double bx = -0.140546;
    const double ay = -2.006687;
    const double by =  0.087909;
    const double az = -0.856645;
    const double bz =  2.152915;

    const double xn = (center_x - cx) / fx;
    const double yn = (center_y - cy) / fy;
    const double r2 = xn * xn + yn * yn;

    const double x = tx_m * 1000.0;
    const double y = ty_m * 1000.0;
    const double z = tz_m * 1000.0;

    CompensatedCamPoint out;
    out.x_mm = x + (ax * xn + bx * xn * r2) * z;
    out.y_mm = y + (ay * yn + by * yn * r2) * z;
    out.z_mm = z * (1.0 + az * r2 + bz * r2 * r2);
    return out;
}
```

## 10. 相机移动后的问题

用户移动了相机位置，只给了一个新 Tag 点和新的 `Q1/Q2/Q3`。

尝试复用旧 `T_shell_cam_corrected` 失败，误差超过 1m。

进一步检查：

```text
旧 Q1Q2 = 86.77mm
旧 Q1Q3 = 48.99mm
旧 Q2Q3 = 128.69mm

新 Q1Q2 = 90.01mm
新 Q1Q3 = 43.94mm
新 Q2Q3 = 129.74mm
```

短边 `Q1Q3` 从 48.99mm 变成 43.94mm，变化约 5mm。

结论：

```text
1. 理论上 T_shell_cam 应该复用。
2. 但当前 Q1/Q2/Q3 三点间距太短，建 Shell 坐标系很敏感。
3. 特别是用 Q1->Q3 这种 40~50mm 短边建主轴，会放大全站仪测量误差。
4. 如果只是相机纯平移，应优先固定旋转 R，只用 Q1 做平移验证。
```

但纯平移模型在 6m 点上仍有约 212mm 误差，说明补偿模型可能不适合从 1~3m 直接外推到 6m。

结论：

```text
补偿模型在当前近距离数据有效；
跨相机位置/远距离使用仍需新距离样本验证。
```

## 11. 内参标定方向

用户手头有棋盘格和全站仪。

当前最重要的不是换相机，而是集中解决边缘误差。

重新标定 K/D 时：

```text
1. 标定图片必须覆盖左边缘、右边缘、上边缘、下边缘和四角。
2. 不要只拍中心。
3. 标定后不能只看平均重投影误差，要用全站仪 + Tag 验证边缘误差。
4. 优先测试 OpenCV 标准 5 参数。
5. 若边缘仍差，测试 rational model: k1~k6。
6. 若仍不够，再做 residual map / 等效补偿。
```

判断 K/D 是否准：

```text
平均重投影误差 < 0.5 px 比较理想
但最终以全视场 Tag 世界坐标误差为准
```

用户关注的核心目标：

```text
边缘误差能不能压到 30mm 内。
```
