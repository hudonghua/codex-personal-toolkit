正常分支里先调用：

这个函数把 `hand_logic()` 生成的 `NO1_Up/Down/Left/Right_ramp_OUT` 混合成四个履带电流：

它支持前进/后退叠加左右转。慢档会先把基础行走电流压到 70%，转弯时再按快档 75% 或慢档 90% 二次限制，然后用左右差速实现转向。

也就是说本地模式不允许依附逻辑，不使用遥控小臂/转台动作。

遥控超时分支：

`gCommOvDly` 在 1ms 定时器里累加，遥控包解析成功后清零。所以 `>2000` 大约是 2 秒无有效遥控通信。超时后清依附、清斜坡、清动作、清清洗盘命令和回转输出。

遥控正常分支：

```c

{
    AppRemoteModeLogic();

普通遥控动作在这里计算：

```text
前臂左/右 = retain比例 * 遥控按键
前臂伸/缩 = retain电流 * 遥控按键
小臂伸/缩 = retain电流 * 遥控按键互锁
转台左/右 = NO2 左右斜坡输出
小臂动作时打开小臂减压阀和保持阀
转台动作时打开转台减压阀
然后调用 pitch_mode() 处理依附
```

`pitch_mode()` 是依附闭环的核心。它只允许一个依附方向成立：

```text
1 = 上依附
2 = 下依附
3 = 左依附
4 = 右依附
```

如果同时按了多个依附方向，或者当前依附方向切换到另一个方向，会清依附内部状态并退出。

依附控制按压力分三段：

```text
压力 < 下限:
  认为压力不足
  减压阀输出恢复到对应初始值
  给补偿动作电流
  设置滤波计数

压力 > 上限:
  认为压力过高
  减压阀输出按步长递减
  给补偿动作电流
  设置高压递减计数

下限 <= 压力 <= 上限:
  如果滤波/高压计数还没结束，继续维持一段时间
  否则不再主动补偿
```

四个方向对应关系：

```text
上依附:
  压力 = SmallArm_Extend_Press_MPa
  动作 = SmallArmExtendCurrent_PWM = UpPitchMakeupCurrent_retain
  减压阀 = SmallArmPropValvePct_PWM
  保持阀 = O_HoldValve_DO

下依附:
  压力 = SmallArm_Retract_Press_MPa
  动作 = SmallArmRetractCurrent_PWM = DownPitchMakeupCurrent_retain
  减压阀 = SmallArmPropValvePct_PWM
  保持阀 = O_HoldValve_DO

左依附:
  压力 = Left_Rotate_Press_MPa
  动作 = TurretLeftCurrentCmd_PWM = LeftPitchMakeupCurrent_retain
  减压阀 = TurntableRotatePropValvePct_PWM

右依附:
  压力 = Right_Rotate_Press_MPa
  动作 = TurretRightCurrentCmd_PWM = RightPitchMakeupCurrent_retain
  减压阀 = TurntableRotatePropValvePct_PWM
```

有一个细节：如果小臂手动动作正在发生，`pitch_mode()` 会直接 `return 1`，不覆盖小臂手动动作；如果转台手动动作正在发生，左/右依附也会让位给手动转台动作。

`work_logic()` 后半段有两个统一后处理。

保持阀延时：

```c
if ((SmallArmExtendCurrent_PWM > 0 || SmallArmRetractCurrent_PWM > 0) && SmallArmPropValvePct_PWM > 0)
```

小臂有动作并且小臂减压阀有输出时，保持阀会延时 50 个 10ms 周期后再打开，也就是约 500ms。前 500ms 即使前面逻辑置了 `O_HoldValve_DO = 1`，这里也会强制压成 0。

行走制动阀延时：

```text
只要 NO1 任一行走/转向斜坡输出 > 0:
  O_BrakeValve_DO = 1
  TravelBrakeOffDelayCnt10ms = 50

行走停止后:
  继续保持制动阀打开约 500ms
  计数结束后关闭
```

最后是倾角保护：

```c
if (TiltOver3Deg_DI && !R119_LimitRelease_DI && !free_flags)
```

车体倾斜超过 3 度，并且本地限位解除和遥控限位解除都没有开启时，会清依附、清斜坡、清动作输出、清制动阀和制动延时。它不等同于急停，主要清运动输出。

**屏幕与遥控发射机显示协议**

这里说的是控制器通过 UART1 回发给遥控发射机屏幕的数据，发送函数是 [Uart1_WL_Send](<F:\工作\AI模型\c除锈机器人\codex\KX_LCD70_200_10AI\Src\App_comm.c:228>)。它只在主循环收到有效遥控包后回发：

```c
if(gUart1RcvFlg == 1)
{
    Uart1_WL_Rcv();
    if(gWLRcvOk >= 1)
    {
        gWLRcvOk = 0;
        Uart1_WL_Send();
    }
}
```

协议帧结构是：

```text
Byte0  0x5A
Byte1  0xA5
Byte2  Len
Byte3  Type
Byte4  Data0
Byte5  Data1
Byte6  Data2
Byte7  Data3
Byte8  Data4
Byte9  Data5
Byte10 Data6
Byte11 Data7
Byte12 XOR
Byte13 CHK
```

注意：代码里的 `Len` 不是“纯数据长度 8”，而是当前写入位置 `i`。对于标准 8 字节数据帧，`Len = 12`，也就是 XOR 所在的下标。校验循环是：

```c
for(j = 2; j < i; j++)
{
    vXor ^= gWLSStr[j];
    vChk += gWLSStr[j];
}
```

也就是从 `Len` 字节开始，一直到 `Data7` 为止参与校验，不包含帧头、不包含 XOR、不包含 CHK。

Type 轮询机制：

```c
static unsigned char vCh = 1;
gWLSStr[i++] = vCh;
...
vCh++;
if(vCh >= 3)
    vCh = 0;
```

所以当前实际发送顺序是：

```text
1 -> 2 -> 0 -> 1 -> 2 -> 0 ...
```

也就是说代码当前只轮询 `Type 0/1/2`，不会发送 `Type 3`。虽然 `case 3` 代码还在，但 `if(vCh >= 3) vCh = 0` 把它跳过了。文档里写的 Type 3 目前更像预留或历史设计。

`Type 0` 是主页面压力 + 状态，数据由 [AppFillRemoteMainPageData](<F:\工作\AI模型\c除锈机器人\codex\KX_LCD70_200_10AI\Src\App_comm.c:35>) 填。它内部还会在两组压力之间交替：

```text
Data6 = 1:
  Data0~1 小臂伸压力 = SmallArm_Extend_Press_MPa * 10
  Data2~3 小臂缩压力 = SmallArm_Retract_Press_MPa * 10
  Data4~5 变幅压力   = Boom_Cylinder_Press_MPa * 10

Data6 = 2:
  Data0~1 油泵压力   = OilPump1_Press_MPa * 10
  Data2~3 左回转压力 = Left_Rotate_Press_MPa * 10
  Data4~5 右回转压力 = Right_Rotate_Press_MPa * 10
```

低字节在前，高字节在后：

```c
DataLow  = value & 0xff;
DataHigh = value >> 8;
```

显示端解析时：

```c
value = DataLow + DataHigh * 256;
pressure_MPa = value / 10.0;
```

`Type 0` 的 `Data7` 是状态字节：

```text
bit0 滤器报警：oil_in_filter_alarm || oil_back_filter_alarm，且会叠加 buf7 原始状态
bit1 倾角报警：TiltOver3Deg_DI
bit2 左依附：Left_pitch_flags
bit3 右依附：Right_pitch_flags
bit4 上依附：Up_pitch_flags
bit5 下依附：Down_pitch_flags
bit6 风机状态：bMotor_flags
bit7 限位解除：free_flags
```

这里有个小点：状态字节初值是 `buf7`，再 OR 上这些状态。因此 `buf7` 里已有的位也会透传到屏幕，显示端最好把它当“控制器状态综合字节”，不要以为只有这几个变量。

`Type 1` 发送 `gRunInfo.vRcvs[8..15]`，当前代码没有看到稳定填充，基本是保留数据。

`Type 2` 发送 `gRunInfo.vRcvs[16..23]`，注释写的是 AD1~AD4 或电磁阀电流类数据。但这轮阅读里没有看到它们在当前主逻辑里被完整填充，所以显示端如果要用 Type 2，需要先确认这些数组元素在哪里赋值。

`Type 3` 理论上发送 `gRunInfo.vRcvs[24..31]`，但如上所说，当前轮询不会进入 `case 3`，除非把 `if(vCh >= 3)` 改为 `if(vCh >= 4)`。

协议示例，假设发 Type 0 第一组压力：

```text
5A A5 0C 00 LL HH LL HH LL HH 01 STATE XOR CHK
```

例如小臂伸 12.3MPa、小臂缩 10.0MPa、变幅 8.5MPa：

```text
压力编码:
12.3MPa -> 123 -> 0x007B -> 7B 00
10.0MPa -> 100 -> 0x0064 -> 64 00
8.5MPa  -> 85  -> 0x0055 -> 55 00

帧主体:
5A A5 0C 00 7B 00 64 00 55 00 01 STATE XOR CHK
```
