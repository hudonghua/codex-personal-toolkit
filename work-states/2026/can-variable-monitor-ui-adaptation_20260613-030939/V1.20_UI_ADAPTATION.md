# V1.20 UI Adaptation Notes

## Problem

On this computer at DPI 144 / 150% scaling, the app was not presenting the same as on another computer. The visible symptom was not only the `演示` button; several toolbar and panel areas could be squeezed because the app mixed fixed pixel sizes with persisted splitter widths from `monitor_profile.json`.

## Main fixes

1. `MainForm` now records effective UI DPI and uses `Ui(...)` helpers for fixed heights, button sizes, splitter widths, list/grid columns, and flowchart geometry.
2. `MonitorProfile` now contains `UiDpi`; saved splitter and column widths are scaled when loaded on a different DPI.
3. Loading a profile no longer immediately saves it back. This prevents startup layout initialization from overwriting a good profile with temporary or invalid splitter values.
4. Layout persistence now checks whether left, monitor, and program panes are wide enough before saving splitter/column metrics.
5. `ClampRightSplitter` enforces a useful minimum for the `2 看数值` pane and leaves enough room for `3 看程序`.
6. The app starts maximized by default because this UI is a dense three-pane engineering workbench.
