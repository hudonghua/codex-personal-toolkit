## Latest Field Finding

PCANView showed:

- `0x10 FF FE FD DF EF`
- `0x15 04 01 5A 6B 7C`
- `0x16 00` increasing rapidly
- Download progress stayed at `0%`

Interpretation:

- `0x15` proves the boot handshake had succeeded.
- Repeated `0x16 00` is the controller bootloader rejecting a control command, not a host-side progress state.
- The most likely trigger was the upper computer probing `0x11 82 01` fast mode against older bootloader firmware that does not support the `0x82` command.

## V1.20 Code Changes

File: `CanFirmwareDownloader.cs`

- Disabled fast-mode probing by default with `EnableFastModeProbe = false`.
- Treats `0x16 00` as a failed boot control ACK instead of waiting forever.
- Clears pending protocol frames before each boot control command.
- Stops storing `0x16` frames in the pending queue, so stale ACKs cannot satisfy the next command.
- While waiting for boot handshake, repeated `0x16 00` now triggers `0x10` resync instead of staying stuck.

File: `MainForm.cs`

- Bumped upper-computer version to `V1.20`.

## SYS Hot-Unplug Note

Windows logs showed a real blue-screen reboot after unplugging SYS during download:

- BugCheck: `0x0000007E`
- Exception: `0xC0000005`
- Dump: `C:\WINDOWS\MEMORY.DMP`
- SYS driver package: `usbcan.inf`
- SYS driver version: `2013/10/16 5.0.2.0`

Conclusion:

- This is kernel-driver level failure, not a normal C# exception.
- Software can reduce the trigger window by avoiding stuck download states and excessive retries, but cannot fully catch a kernel blue screen caused by the old SYS driver.
