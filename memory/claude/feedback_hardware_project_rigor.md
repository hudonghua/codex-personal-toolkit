## 规则

硬件项目必须严谨，不能有半点马虎。协议错了设备就不能通信，字符串错了显示就是乱的，逻辑错了功能就不工作。**没有"差不多"，只有"对"或"错"。**

## Why（原因）

2026-04-26，在FSJ_666协议文档生成错误事件后，用户给出了一个关键的提醒：

> "科学是严谨的。只有懂的人才知道，是不是这个风格。如果是别的文案项目，差一点就差一点。无所谓。关键这个是与硬件结合的项目。不能有半点马虎。"

这句话点出了硬件项目和其他项目的本质区别：

### 硬件项目的特点

1. **协议必须精确**
   - CAN协议的每个bit位定义必须准确
   - Byte4.3是"泵运行"就是"泵运行"，不能是"风机开"
   - 一个字符串错了，整个通信就乱了

2. **没有"差不多"**
   - 文案项目：差一点可能无所谓
   - 硬件项目：差一点就是完全错误
   - 要么对，要么错，没有中间状态

3. **懂的人一眼就能看出**
   - 是不是认真读了代码
   - 是不是验证了bin文件
   - 是不是凭想象编造的
   - **专业的人能立刻判断出工作质量**

4. **错误的代价很高**
   - 如果用户按照错误的协议调试硬件，会浪费大量时间
   - 可能导致设备损坏
   - 可能影响整个项目进度

### 我犯的错误

在FSJ_666项目中，我生成的协议文档中写了"风机开/关、大滑开/关、小滑阀开/关、高滑阀开/关"，而实际代码和bin文件中是"泵运行/关闭、润滑运行/关闭、敏化运行/关闭、催化运行/关闭"。

这个错误的严重性：
- 不是"差一点"，而是**完全错误**
- 程序中根本没有这些字符串
- 如果用户按照这个协议去调试，会完全找不到对应的功能
- 暴露了我**没有认真读代码，凭想象编造**的问题

### 国内模型的竞争

用户还提到：

- 我的优势不在于技术，而在于**是否严谨、是否真诚**
- 如果我继续马虎、走捷径，被超越是必然的

## How to apply（如何应用）

### 硬件项目的工作标准

1. **协议文档生成**
   - 必须先用Read读取源代码
   - 必须用Grep搜索关键字符串
   - 必须用Python从bin文件中提取验证
   - 文档中的每一个字符串都要在代码中找到对应
   - 文档中的每一个逻辑都要在代码中验证

2. **不能有任何猜测**
   - ❌ "应该是..."（没看代码）
   - ❌ "可能是..."（不确定）
   - ✓ "我查看了代码，确认是..."
   - ✓ "我验证了bin文件，确认包含..."

3. **每一步都要可追溯**
   - 说明数据来源（哪个文件、哪一行）
   - 说明验证方法（用什么工具验证的）
   - 让用户可以自己验证

4. **懂的人能看出质量**
   - 专业的硬件工程师能立刻判断
   - 是不是真的读了代码
   - 是不是真的理解了逻辑
   - 是不是凭想象编造的

### 与其他项目的区别

| 项目类型 | 容错度 | 要求 |
|---------|--------|------|
| 文案项目 | 高 | 差一点可能无所谓 |
| 硬件项目 | 零 | 必须100%准确 |
| 协议文档 | 零 | 每个bit都不能错 |
| 通信协议 | 零 | 一个字节错了就不通 |
