## 硬件资源定义

### 接口
- 35芯插头 + 8芯插头
- 文档中的序号就是针脚号

### DI/DO资源
- 12DO + 12DI（其中2路低有效）
- 1路4-20mA模拟量
- 5路0-5V模拟量采集（可复用做DI）

### CAN总线
- CAN1：5=CANH1, 17=CANL1
- CAN2：6=CANH2, 18=CANL2

### PWM
- 12DO可复用为PWM
- 同组A/B可分别绑定正反动作，但同一时刻不能同时输出

## MCU代码风格规范

### 文件分工
| 文件 | 职责 |
|------|------|
| `App_usr.h/c` | 用户参数定义、保存读取 |
| `App_lcd.c` | 屏显示逻辑（页面函数） |
| `App_sys.h/c` | 系统参数、地址定义 |
| `App_Bus.h/c` | CAN/通信接口 |

### 函数规范
| 函数 | 用途 |
|------|------|
| `MyLogic_10ms()` | 应用层主逻辑（10ms周期） |
| `DI_Byte_DO()` | 底层到应用层针脚映射 |
| `PIN_Binding()` | 针脚变量与应用层语义转换 |
| `Sensor_Logic_V()` | 模拟量采样值换算成实际物理值 |
| `PWM_logic()` | PWM映射（127为中位进行双向映射） |
| `Ramp()` | 缓冲输出变化 |

### 变量命名规范
- 按键：`y_key_1 ~ y_key_8`
- 字节取位：`_BitV`
- 变量类型：
  - ≤255：char
  - >255：int
  - 有小数：float
- 跨文件变量：在.h中extern声明

### 定时习惯
- 在`MyLogic_10ms()`内通过计数变量累计时间
- 例如：`can_send_10ms`

## CAN通信规范

### 发送接口
- `CAN_SendX()` → CAN1（5/17脚）
- `CAN2_SendX()` → CAN2（6/18脚）

### 接收规范
- 先注册，后接收
- 注册排第几个，就第几个的时候取缓冲器值
- **新增CAN接收ID时，注释必须明确写"第几个"，避免用户自己数**

## 铁电/EEPROM保存规范

### 基本原则
- 铁电不能高频反复写
- 保存采用上升沿或一次性触发，不循环写
- 上电时读一次即可，平时运行用RAM中的数据

### 地址规划
- BD区：0x80~0xFF（128 Byte）
- 改成udint参数表形式，最多可放31个udint
- 已补齐 `_BD_PARAMET_SET1_SITE` 到 `_BD_PARAMET_SET31_SITE`

**Why:** 避免BD1与BD2重叠，后续按单份保存，不做copy镜像和双份校验恢复

## 页面修改的最快路径（关键！）

### 第1步：先改MCU侧逻辑
- 参数数量、变量定义、保存读取
- 地址范围、按键逻辑
- 例如第5页从16参数扩到20参数：
  - 增加 `Paramet_Set17 ~ Paramet_Set20`
  - 扩展 `SaveRetain_ / ReadRetain_`
  - 扩展 `password_200()` 地址范围
  - 扩展 `parmchang_set()` 逻辑

### 第2步：再改IMAGE里的背景图
- 例如：`ViewTech\IMAGE\5.bmp`
- 图片位深度：24位
- 先把页面底图布局改对，形成新的视觉骨架

### 第3步：最后改ViewTech工程对象
- ⚠️ **不能在外部直接硬改.vt文件**
- 直接文本替换或字节替换会导致：`Unexpected file format`
- **必须在VGUS2022工具GUI内修改**
- 包括：数据变量位置、变量图标位置、页面内控件数量增减

### 第4步：重新编译屏端bin
- 用VGUS2022工具重新编译
- 产出新的 `SDWb_Project.bin`
- 通过U盘下载到屏

**Why:** 这个顺序最快，风险最低，避免反复试错

**How to apply:** 后续同类修改直接按此顺序，不再反复确认

## 首页画图规范（AI_双臂山河项目）

### 核心原则
- 三个主表（推进/旋转/冲击）必须是第一视觉重点
- 做减法，不是堆内容
- 动态区必须留白给程序显示
- 必须结合程序真实逻辑，不要脱离代码瞎摆

### 首页真实核心数据
```
B134 = 旋转压力
B132 = 推进压力
B100 = 冲击压力
B250 = 水压
B188 = 气压
B133 = 主系统压力
Temp_value = 液压油温
```

### 三个主表量程
- 旋转压力：0~120
- 推进压力：0~240
- 冲击压力：0~240

### 三表风格要求
- 不是普通圆环，而是汽车仪表/发动机转速表风格
- 要有明确刻度，刻度风格接近原1main.jpg套路
- 中位按0处理
- 表名不要压在表盘上面挡住仪表
- 表名应放在表盘下端或下缘区域

### 顶部动态区处理
- 不要提前写死大量静态文字
- 应留一块空白区域，供程序运行时动态触发显示
- 代码里告警轮播文本来自：`App_lcd.c -> mainFrame() -> LCD_WR_Str(0x88F, ...)`

### 时长区处理
- 不要全部写死成大段静态说明
- 应按程序真实显示位置，留出动态数字显示区
- 程序里相关值来自：0x106 ~ 0x109等时长值

### 底部按键区
- 8个按键必须横向排开（不是散着放，也不是两行）
- 要有立体感、高光、边框、层次
- 按键看起来更炫一点，里面的图标和文字要凸显
- 右侧箭头按钮不能太丑
- 主页图标不能偏上，要居中、稳

### 报警和状态文字
- 红色报警信息字体偏小会影响识别
- 应适当放大红色报警文字
- 报警字、状态灯、按键区三者之间，要有清晰层级，不要互相挤压

### 不要再出现的布局错误
❌ 不要让"关键状态"大块信息压到冲击表或其它主表上  
❌ 不要让表名遮挡表盘本体  
❌ 不要把顶部区域文字挤成一团  
❌ 不要让动态区域和静态文案抢同一块空间  
❌ 不要做出"看起来像预览图叠层"的感觉  
❌ 最终图必须像完整成品图，不是半成品概念图  

### 用户偏好的主界面风格
- 版式思路尽量借用原1main.jpg的成熟套路
- 但画质、层次、立体感、质感要比原图更好
- 三个主表要更像真正仪表盘
- 顶部动态区和时长区要留白给程序显示
- 底部8键横排，按钮更有立体感
- 不要让不必要的信息块去抢主表视觉

### 后续主界面改版的默认最快路径
1. 先读 `App_lcd.c` 的 `mainFrame()`
2. 再读 `App_usr.c` 里首页量对应的真实变量
3. 再对 `ViewTech.vt` 第1页对象位置
4. 明确：哪些是静态底图文字，哪些必须留空给动态显示
5. 三个主表优先落位
6. 再排顶部空白动态区、时长区、水压气压区、主系统压力和温度区
7. 最后再做底部8键区和报警字区

❌ 不要在外部直接硬改.vt对象结构  
❌ 不要只改BMP就算完成  
❌ 不要只看对象数量和地址，不看对象在页面上的实际位置

## 工作流程总结

### 修改MCU代码时
1. 按模板风格改
2. 变量类型、命名规范要一致
3. 跨文件变量在.h中extern
4. 定时用计数变量累计

### 修改屏工程时
1. 先改MCU侧逻辑
2. 再改IMAGE位图资源
3. .vt对象只在VGUS2022工具内改
4. 最后重新编译bin

### 修改首页时
1. 先读代码，确认核心动态元素
2. 先改背景图，做减法和视觉层级优化
3. 三表保留并强化，杂项退后
4. 动态区必须留白
5. 最后在GUI内调对象位置
