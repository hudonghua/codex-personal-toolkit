## 规则

生成协议文档或技术文档时，**必须先读取并分析实际的源代码和二进制文件**，绝对不能凭想象、猜测或模糊记忆生成内容。

## Why（原因）

2026-04-26发生了严重的错误：为FSJ_666项目生成的HTML协议文档中，错误地将装药模式的状态位描述为"风机开/关、大滑开/关、小滑阀开/关、高滑阀开/关"，而实际代码和bin文件中使用的是"泵运行/关闭、润滑运行/关闭、敏化运行/关闭、催化运行/关闭"。

这个错误导致了连锁反应：
1. 用户根据错误的HTML协议要求核对代码
2. 误判代码需要修改
3. 在修改过程中破坏了App_lcd.c源文件（编码问题）
4. 从备份恢复后又多次修改代码逻辑
5. 浪费了大量时间和精力

**根本原因**：生成HTML时没有先读取实际代码，而是凭猜测或错误记忆生成的。程序中根本没有"风机、大滑、小滑、高滑"这些字符串。

## How to apply（如何应用）

### 生成协议文档的正确流程

1. **先读取源代码**
   - 使用`Read`工具读取相关的.c/.h文件
   - 使用`Grep`工具搜索关键字符串和逻辑

2. **验证二进制文件**
   - 使用Python从bin/hex文件中提取实际的字符串
   - 对比源代码和二进制文件，确保一致

3. **分析代码逻辑**
   - 理解每个bit位的判断逻辑
   - 理解显示条件（=1显示什么，=0显示什么）
   - 理解是否有辅助字节（如Byte6）

4. **生成文档**
   - 基于实际代码生成协议文档
   - 文档中的每一项都要有代码依据

5. **交叉验证**
   - 生成后立即用bin文件验证
   - 确保文档描述的字符串在bin文件中都能找到

### 验证顺序

```
源代码（最权威）→ bin文件（验证）→ 协议文档（描述）
```

不能反过来：协议文档 → 修改代码

### 关键原则

- ❌ 不能凭想象生成协议
- ❌ 不能凭模糊记忆生成协议
- ❌ 不能套用其他项目的协议
- ✓ 必须先读取实际代码
- ✓ 必须验证bin文件
- ✓ 文档的每一项都要有代码依据
- ✓ 如果协议文档和代码不一致，首先怀疑协议文档

### 适用场景

- 生成CAN协议文档
- 生成通信协议文档
- 生成接口定义文档
- 生成任何基于代码的技术文档
