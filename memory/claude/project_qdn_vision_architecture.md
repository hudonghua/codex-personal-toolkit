## 一、视觉技术方案

### 硬件配置

**相机**：
- 工业相机，固定在车体上
- 视野覆盖推进梁末端
- 分辨率 >= 1280×720
- 帧率 >= 30fps

**四点光源**：
- 4个LED点光源
- 安装在推进梁末端附近
- 形成刚体（相对位置固定）
- 需要测量四点相对坐标

**工控机**：
- 运行QT + OpenCV
- 多核CPU >= 2GHz
- 内存 >= 4GB

---

### 算法流程

```
1. 相机采集图像（10-30ms）
2. 图像传输（5-20ms）
3. 图像预处理（5-10ms）
4. 识别四个光点（10-30ms）
5. solvePnP位姿解算（5-15ms）
6. 坐标变换（5-10ms）

总延迟：40-115ms
典型值：60-80ms
频率：12-16Hz
```

## 二、实时性分析

### 与控制器对比

```
控制器：10ms周期，100Hz
视觉：60-80ms周期，12-16Hz
对比：视觉比控制器慢 6-8倍
```

---

### 不同工况下的实时性

**慢速微调（< 20mm/s）**：
- 60ms延迟 → 位置变化 < 1.2mm
- 实时性足够

**快速移动（100-200mm/s）**：
- 60ms延迟 → 位置变化 6-12mm
- 视觉数据滞后，不能直接用

**钻孔作业（臂架不动）**：
- 60ms延迟 → 无影响
- 实时性完全足够

## 三、三层架构优化

### 架构调整（关键决策）

**原方案（不太好）**：
```
控制器：
- 第一层：正运动学
- 第二层：坐标变换
- 第三层：视觉融合 ← 在控制器上
- 发送给电脑显示

问题：
- 控制器要处理视觉数据
- 控制器要做融合算法
- 增加控制器负担
```

**新方案（更好）**：
```
控制器：
- 第一层：正运动学
- 第二层：坐标变换
- 发送H1给电脑

电脑（QT）：
- 视觉计算 → H2
- 接收控制器的H1
- 第三层：融合 → H_final ← 在电脑上
- 显示H_final

优点：
- 控制器只管运动学和控制
- 电脑算力强，做融合更合适
- 显示在电脑上，融合也在电脑上
- 职责清晰，开发灵活
```

---

### 职责分工

**控制器职责**：
- 第一层：正运动学（设备坐标系）
- 第二层：坐标变换（大地坐标系）
- 发送H1给电脑
- 专注实时控制（10ms周期）

**电脑职责**：
- 视觉处理（图像采集、识别、solvePnP）
- 接收控制器的H1
- 第三层：融合算法（自学习偏置）
- 界面显示（炮孔图、偏差、状态）
- 数据记录和分析

## 四、电脑端融合算法

### 自学习偏置算法

```python
class VisionFusion:
    def __init__(self):
        self.offset = np.array([0.0, 0.0, 0.0])
        
    def update(self, h1_from_controller, h2_from_vision, vision_valid):
        if vision_valid:
            delta = h2_from_vision - h1_from_controller
            delta_norm = np.linalg.norm(delta)
            
            # 异常检测
            if delta_norm < 50.0:
                # 自学习偏置（滑动平均）
                self.offset = 0.9 * self.offset + 0.1 * delta
        
        # 应用偏置
        h_final = h1_from_controller + self.offset
        return h_final
```

---

### 算法特点

**自学习**：
- 视觉有效时：持续学习更新offset
- 视觉失效时：使用最近学习的offset
- 越跑越准

**平滑降级**：
- 视觉失效不会突变
- 用最近学习的offset补偿
- 平滑过渡

**异常检测**：
- 排除快速移动时的滞后（delta > 50mm）
- 排除视觉异常数据
- 保证学习质量
