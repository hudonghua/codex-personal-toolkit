## 规则

基于用户多年嵌入式硬件项目经验总结的编码规范和命名规则。

## 核心原则（来自 default_ai_codex_model_memory.md）

### 0. 全局工作原则（2026-04-26）

1. 严格基于当前工程里的实际代码、实际数据、实际文件状态工作
2. 记忆、旧结论、历史记录只作导航和辅助，不代替当前事实
3. 不确定时先验证，不要猜测
4. 每一步都必须有依据
5. 涉及中文、老编码、ANSI / Default 编码源文件时，必须先确认编码和文件实况
6. 禁止因转码判断不清而把声明区、函数体或整体文件改坏
7. 一旦出现变量声明区丢失、文件截断等情况，必须立刻停手，先恢复
8. **变量声明和命名默认沿用用户习惯**：
   - 协议层按 `_DI` / `_DO` / `_AI` 区分
   - 掉电保持参数显式带 `_retain`
   - 应用层变量优先使用可读业务语义名

### 1. 代码风格原则

9. 控制器/HMI 这类结合硬件的程序，默认尽量不要新引入：
   - `enum`
   - 复杂数组索引
   - 指针层转发
   - 结构体层层取成员
   - 优先用朴素、直接、可全局搜索的变量和逻辑判断

10. 采用上条风格的主要原因：
    - 让 `Keil` 全局搜索能直接追踪变量生命周期
    - 降低排查硬件联动问题时的阅读成本

11. 这类项目默认不追求"大型软件式"的抽象、内存技巧或高级语法结构
    - 真正核心是把硬件场景经验、联动条件和控制动作用最直白的代码表达清楚

12. 如果底层接口、芯片库、协议缓冲区或现有工程结构已经必须使用数组、指针、结构体：
    - 尽量把复杂性留在边界层
    - 应用层不要继续扩散间接访问

## 变量命名规范

### 协议层变量

| 类型 | 命名规则 | 示例 |
|------|---------|------|
| 数字输入 | `xxx_DI` | `valve_open_DI` |
| 数字输出 | `xxx_DO` | `pump_run_DO` |
| 模拟量输入 | `xxx_AI` | `pressure_AI` |
| 掉电保持参数 | `xxx_retain` | `calibration_value_retain` |

### 应用层变量

- **优先使用可读业务语义名**
- 不要用 `a`、`b`、`temp` 这种无意义名称
- 变量名直接表达业务含义

**示例：**
```c
// ✓ 好 - 业务语义清晰
uint8_t valve_open_signal;
uint8_t pump_run_status;
float pressure_value;

// ✗ 避免 - 无意义
uint8_t a;
uint8_t temp;
float val;
```

### 按键命名

```c
y_key_1 ~ y_key_8
```

### 变量类型选择

| 数值范围 | 类型 |
|---------|------|
| ≤255 | `char` 或 `uint8_t` |
| >255 | `int` 或 `uint16_t` |
| 有小数 | `float` |

### 字节取位

使用 `_BitV` 宏：
```c
if(_BitV(gRunInfo.vRcvWL[4], 1))
{
    // bit 1 为 1
}
```

### PWM 变量命名

当 DO0-DO7 用作比例量时：
```c
DO_0_PWM ~ DO_7_PWM  // 应用层使用
```

不直接使用 `DO_H2`、`DO_H3` 作为比例量变量。

## 函数命名规范

### 常见函数职责

| 函数名 | 职责 |
|--------|------|
| `MyLogic_1ms()` | 1ms 周期任务 |
| `MyLogic_10ms()` | 10ms 周期任务（应用主逻辑） |
| `DI_Byte_DO()` | 底层点位映射 |
| `PIN_Binding()` | 针脚变量与应用层语义转换 |
| `Sensor_Logic_V()` | 模拟量采样值换算成实际物理值 |
| `PWM_logic()` | PWM 映射 |
| `Ramp()` | 缓冲输出变化 |
| `work_logic()` | 动作逻辑 |
| `Disp_main()` | 界面输出 |

### 函数位置规则

- `MyLogic_10ms()` 应保持在文件后部
- 确保其调用的函数和变量已在前文声明或定义
- 避免语法和隐式声明问题

## 文件分工规范

| 文件 | 职责 |
|------|------|
| `App_usr.h/c` | 用户参数定义、保存读取 |
| `App_lcd.c` | 屏显示逻辑（页面函数） |
| `App_sys.h/c` | 系统参数、地址定义 |
| `App_Bus.h/c` | CAN/通信接口 |

## CAN 相关规范

### CAN ID 命名规则

变量名与 CAN ID 的映射关系：

```
变量名     CAN ID
EA05   →  0x2EA.05
```

这种命名方式将 CAN ID 编码到变量名中，便于追踪。

### CAN 接收规范

1. 先注册 CAN 接收
2. 再按第几个缓冲区取数
3. **新增 CAN 接收 ID 时，注释必须明确写"第几个"，避免用户自己数**

**风险：** 如果后续改 CAN 接收 ID，只改注册、不改业务判断，很容易出现缓冲区错位。

## 定时习惯

在 `MyLogic_10ms()` 内通过计数变量累计时间：

```c
static uint16_t can_send_10ms = 0;

can_send_10ms++;
if(can_send_10ms >= 10)  // 100ms
{
    can_send_10ms = 0;
    // 执行 100ms 周期任务
}
```

## 跨文件变量

在 `.h` 中 `extern` 声明：

```c
// App_usr.h
extern uint8_t valve_status;
extern float pressure_value;
```

在 `.c` 中定义：

```c
// App_usr.c
uint8_t valve_status = 0;
float pressure_value = 0.0f;
```

## 工程修改纪律（2026-04-25）

1. 用户指定修改某个函数时，必须严格限制在该函数块和最小必要的声明范围内
2. 不得用整文件脚本替换跨函数重写
3. 对 C 工程中的大文件，优先采用"回读函数边界 → 局部修改 → 立即编译"的闭环
4. 不得在未编译验证前连续叠加多个任务
5. 若发现源码出现函数体被切断、全局变量插入函数中间等结构性异常，必须立刻停止
6. 遇到 `.h` 有 `extern`、`.c` 缺定义的问题，应优先做对账补齐
7. 如果已出现大面积链接错误，说明基线已污染，必须回退而不是继续盲补

## 编码处理规则

1. 原目录中同时存在 UTF-8、GBK、GB2312 文本
2. 某些旧 `.txt` 文件如果直接用默认编码打开，可能乱码
3. 读取时建议优先显式指定编码
4. 涉及老编码 C / H 源文件（ANSI / Default 等）时，必须先确认编码
5. 禁止对老编码源码做一刀切的整文件替换、整段声明区清理、批量转码覆盖
6. 一旦出现变量声明区丢失、文件截断、重复块、花括号异常等现象，必须立刻停手

## 编码器角度换算（Arm200A 项目）

### 单圈 16 位编码器

```c
static float Enc16_ToDeg(unsigned short raw, unsigned short zero, float ratio)
{
    int delta;
    
    delta = (int)raw - (int)zero;
    
    if(delta > 32767)
    {
        delta -= 65536;
    }
    else if(delta < -32768)
    {
        delta += 65536;
    }
    
    return ((float)delta) * (360.0f / 65536.0f) * ratio;
}
```

### 调用原则

- `ratio` 只代表传动比系数，不包含方向信息
- 如果方向需要反，在调用处显式取负：

```c
// 正常方向
gArmLeftIn.base_deg = Enc16_ToDeg(base_raw, base_zero_raw, base_ratio);

// 方向反了
gArmLeftIn.base_deg = -Enc16_ToDeg(base_raw, base_zero_raw, base_ratio);
```

## 200A屏项目统一套路（来自除锈机器人项目）

### 应用层架构

1. **应用层总调度集中在 `MyLogic_10ms()`**
2. **`PIN_Binding()`** 只做"固定点位变量 → 应用层语义变量"绑定
3. **`CAN1_IO_pinding()`** 只按119协议发送到119，不混别的逻辑
4. 不要拆散成熟函数调用顺序；若用户指定顺序，严格遵守

### 命名风格（119协议相关）

| 类型 | 后缀 | 示例 |
|------|------|------|
| 119 输入变量 | `_DI` | `valve_open_DI` |
| 119 输出变量 | `_DO` | `pump_run_DO` |
| 119 模拟量变量 | `_AI` | `pressure_AI` |
| 铁电/retain变量 | `_retain` | `calibration_value_retain` |

**重要原则：**
- 应用层别名要让人一看就懂
- 避免到处直接裸用 `gPwm_BD[x].vPwm_Param[i][j]`
- 尽量不用宏做映射，优先用应用层变量 + 周期同步函数
- 若别名来源于铁电/参数区，变量名后面最好带 `_retain`，便于一眼识别

### 手柄参数风格

- `hand_logic()` 中不要直接大量裸用 `gPwm_BD[][]`
- 可以建立可读别名变量
- 别名变量如果来自铁电，后缀带 `_retain`

**示例：**
```c
// ✓ 好 - 可读别名
float FrontArmLeftTurnPct_retain;
float VacuumFanMotorPct_retain;

// ✗ 避免 - 直接裸用
gPwm_BD[0].vPwm_Param[1][2]
```

### 119 比例/开关输出规则

- 0x15A 协议严格按 byte0~byte7 对应 `DO_0_PWM ~ DO_7_PWM`
- **注意：** 某些项目中 DO5/DO6 可能用作开关量（如启动/熄火急停），不是 PWM 业务量

### 页面参数规则

- 每个参数页的显示/选择/加减/默认值/保存逻辑应在对应的 `Frame_XX()` 函数中
- 不要错误地全放到全局同步函数里硬灌
- `SaveRetain_` / `ReadRetain_` 只负责存储链路

## 已经犯过的错误（绝对不能再犯）

### 1. 文件编辑错误

❌ **在 ANSI 编码源文件上使用整文件正则 / 大块文本替换**

后果：
- 变量声明区被截断
- 函数体丢失
- 出现重复块、花括号错位
- `expected a declaration` 等低级错误

✓ **正确做法：**
- 最小范围修改
- 逐点修改策略
- 改完立即检查

### 2. 发现错误后继续修改

❌ **发现截断后没有立刻停手，而是在坏文件上继续改**

后果：
- 错误扩大
- 重复定义、重复赋值块越来越多

✓ **正确做法：**
- 发现异常立刻停手
- 先恢复文件
- 再继续修改

### 3. 逻辑放错位置

❌ **把页面参数逻辑错误放进全局同步函数**

✓ **正确做法：**
- 页面逻辑放在 `Frame_XX()` 函数中
- 全局同步函数只做同步

### 4. 引入不必要的中间变量

❌ **引入中间别名如 `R119_DO0_PropPct` / `R119_DO1_PropPct`**

用户不接受这种风格。

✓ **正确做法：**
- 0x15A 直接使用 `DO_0_PWM ~ DO_7_PWM`

### 5. 一次性做过多结构性替换

❌ **在 `hand_logic()` 可读化过程中，一次性做过多结构性替换**

✓ **正确做法：**
- 先加可读别名
- 再改 `hand_logic()` 引用
- 再小步重构同步函数
- 每一步都要检查有没有残留重复块

## 处理这类项目的默认动作

1. **先浏览当前代码实际状态，再动手**
2. **先做最小范围变更，再编译，再继续**
3. **如果用户说"先不搞"，必须挂起，不要顺手继续做**
4. **若用户说"我手动改好了"，后续以当前文件实况为准，不依赖旧中间状态**
5. **任何涉及老编码大文件的编辑，都要优先选择最小范围、逐点修改策略**

## 总结

用户的编码规范核心思想：
1. **简单直接** - 不追求抽象和高级语法
2. **可追踪** - Keil 全局搜索能看到完整生命周期
3. **业务语义** - 变量名直接表达业务含义
4. **分层清晰** - 协议层/应用层/边界层职责明确
5. **严格修改** - 局部修改，立即验证，不破坏基线
