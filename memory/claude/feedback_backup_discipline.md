# 文件修改备份纪律

**规则：修改文件前必须先备份，改坏时不私自回退；备份按"每文件 3 份轮转"管理**

**⚠️ 适用范围（2026-06-05 用户明确）：备份纪律只针对"代码"。改源码（.c/.h 等）前才需要备份；HTML/文档/分析报告这类产物不需要备份，直接改即可，别建多余的 .bak。**

**Why:** 2026-05-01 发生严重事故：恢复旧备份覆盖了用户的 6 路压力显示代码，导致工作成果丢失。用户非常愤怒。必须建立严格的备份和退档流程。2026-05-27 用户进一步规定：备份要节制，每个 .c 文件只保留 3 份，超过删最旧的，避免备份池膨胀干扰定位。

**How to apply:**

## 1. 修改前必须备份
- 每次修改文件前，先用 `cp`/`Copy-Item` 备份
- **只备份正在改的那个文件**，不要顺手把整个工程的 `.uvprojx`/其他 `.c` 一起备份
- 备份文件名格式：`原文件名.bak_YYYYMMDD_HHMMSS`
- 例如：`App_lcd.c.bak_20260501_143000`
- **每个文件最多保留 3 份备份**，到第 4 份时先删最旧的再写入（详见 § 6）

## 2. 改坏时不私自回退
- 发现改错了，**不要**直接执行 `cp 备份 原文件`
- **不要**私自决定回退到哪个版本
- 必须先告知用户当前状态和可用的备份

**特别重要：以下问题绝对不能退档，必须现场修复：**
- ❌ 大括号 `{}` 作用域问题 → 用 Python 修复大括号
- ❌ 中文乱码问题 → 用 Python 重新写入正确的中文（GBK 编码）
- ❌ 编码问题 → 用 Python 转换编码或修复文本
- ❌ 任何可以通过 Python 脚本修复的问题

**原因：** 退档太快，用户来不及备份当前程序。现场修复可以保留用户的最新工作。

## 3. 退档流程
当需要恢复备份时：
1. **必须先提醒用户**："发现问题，需要退档吗？"
2. 列出该文件当前所有备份（最多 3 份）及时间戳
3. 说明每个备份的内容（如果知道）
4. 询问用户："准备回到哪一份备份？"
5. 等待用户确认
6. 用户确认后才执行恢复命令

**重要：绝对不能随意退档现有程序！必须先征得用户同意！**

## 4. 禁止的操作
- ❌ **不经用户同意就恢复旧备份（最严重！）**
- ❌ **随意退档现有程序**
- ❌ **遇到 {}、中文、编码问题就退档（用户来不及备份！）**
- ❌ 用旧备份覆盖包含新功能的文件
- ❌ 不检查备份日期就盲目恢复
- ❌ 没有备份就修改文件
- ❌ 发现问题后直接退档，而不是先提醒用户
- ❌ 能用 Python 修复的问题却选择退档
- ❌ 改一个 .c 把整个工程或多个无关文件一起备份
- ❌ 让单个文件的备份数量超过 3 份

## 5. 示例

**正确做法：**
```powershell
# 1. 改之前先看池
Get-ChildItem App_lcd.c.bak_*

# 2. 如果已有 3 份，删掉最旧的
# (按 § 6 的规则)

# 3. 写新备份
Copy-Item App_lcd.c "App_lcd.c.bak_$(Get-Date -Format yyyyMMdd_HHmmss)"

# 4. 改文件
```

**错误做法（2026-05-01 的教训）：**
```powershell
# ❌ 直接用旧备份覆盖，丢失了用户的 6 路压力显示代码
Copy-Item App_lcd.c.bak App_lcd.c   # 这个备份是 4 月 30 日的旧版本！
```

## 6. 备份轮转规则（2026-05-27）

**只备份正在改的那个文件。**改 `App_lcd.c` 就只备份 `App_lcd.c`，不要附带 `App_usr.c` 或 `.uvprojx`。

**每个文件最多保留 3 份备份**：池满时（已有 3 份），写入第 4 份之前必须先删除同文件名最旧的那一份。

当前池示例（按时间戳排序）：
```
App_lcd.c.bak_20260525_180000   ← 最旧（要删）
App_lcd.c.bak_20260526_103000
App_lcd.c.bak_20260527_090000   ← 最新
```

下一次再改 `App_lcd.c`：
1. 删除 `App_lcd.c.bak_20260525_180000`
2. 写入 `App_lcd.c.bak_20260527_<HHMMSS>`
3. 池中恒为 3 份

**轮转只在同文件名内**：`App_lcd.c` 和 `App_usr.c` 的备份池互不干扰，各自最多 3 份。

**Why:** 每次改都堆新备份会迅速膨胀，让真正有用的回档点淹没在噪音里。3 份足以覆盖"上一次改前 / 上上次改前 / 更早一份"，再多就是浪费且增加误退档风险。

**示例脚本（PowerShell）：**
```powershell
function Backup-WithRotate {
    param([string]$File, [int]$Max = 3)
    $dir = Split-Path $File -Parent
    if (-not $dir) { $dir = "." }
    $name = Split-Path $File -Leaf
    $existing = Get-ChildItem -Path $dir -Filter "$name.bak_*" |
                Sort-Object LastWriteTime
    while ($existing.Count -ge $Max) {
        Remove-Item $existing[0].FullName
        $existing = $existing[1..($existing.Count-1)]
    }
    $stamp = Get-Date -Format yyyyMMdd_HHmmss
    Copy-Item $File "$File.bak_$stamp"
}
```

## 7. 退档时如何展示给用户
当池里有 3 份时，退档询问应这样展示：

```
我：发现 App_lcd.c 改坏了。当前 3 份备份：
  - App_lcd.c.bak_20260527_090000  (今早改前 — 含你刚加的 XX 功能)
  - App_lcd.c.bak_20260526_103000  (昨天上午改前)
  - App_lcd.c.bak_20260525_180000  (前天晚上 — 早期稳定版)
  准备回到哪一份？

用户：回到 App_lcd.c.bak_20260527_090000

我：好的，恢复后请在 Keil 里 Rebuild 并烧录验证。
```
