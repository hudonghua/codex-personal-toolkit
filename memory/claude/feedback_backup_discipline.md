# 文件修改备份纪律

**规则：修改文件前必须先备份，改坏时不私自回退**

**Why:** 2026-05-01发生严重事故：恢复旧备份覆盖了用户的6路压力显示代码，导致工作成果丢失。用户非常愤怒。必须建立严格的备份和退档流程。

**How to apply:**

## 1. 修改前必须备份
- 每次修改文件前，先用`cp`命令备份
- 备份文件名格式：`原文件名.bak_YYYYMMDD_HHMMSS` 或 `原文件名.bak_描述`
- 例如：`App_lcd.c.bak_20260501_143000` 或 `App_lcd.c.bak_before_pressure_fix`
- 避免覆盖已有备份，每次用新的时间戳

## 2. 改坏时不私自回退
- 发现改错了，**不要**直接执行 `cp 备份 原文件`
- **不要**私自决定回退到哪个版本
- 必须先告知用户当前状态和可用的备份

**特别重要：以下问题绝对不能退档，必须现场修复：**
- ❌ 大括号 `{}` 作用域问题 → 用Python修复大括号
- ❌ 中文乱码问题 → 用Python重新写入正确的中文（GBK编码）
- ❌ 编码问题 → 用Python转换编码或修复文本
- ❌ 任何可以通过Python脚本修复的问题

**原因：**退档太快，用户来不及备份当前程序。现场修复可以保留用户的最新工作。

## 3. 退档流程
当需要恢复备份时：
1. **必须先提醒用户**："发现问题，需要退档吗？"
2. 列出所有可用备份及其时间戳
3. 说明每个备份的内容（如果知道）
4. 询问用户："准备回到哪一份备份？"
5. 等待用户确认
6. 用户确认后才执行恢复命令

**重要：绝对不能随意退档现有程序！必须先征得用户同意！**

## 4. 禁止的操作
- ❌ **不经用户同意就恢复旧备份（最严重！）**
- ❌ **随意退档现有程序**
- ❌ **遇到{}、中文、编码问题就退档（用户来不及备份！）**
- ❌ 用旧备份覆盖包含新功能的文件
- ❌ 不检查备份日期就盲目恢复
- ❌ 没有备份就修改文件
- ❌ 发现问题后直接退档，而不是先提醒用户
- ❌ 能用Python修复的问题却选择退档

## 5. 示例

**正确做法：**
```bash
# 1. 先备份
cp App_lcd.c App_lcd.c.bak_$(date +%Y%m%d_%H%M%S)

# 2. 修改文件
sed -i 's/old/new/' App_lcd.c

# 3. 发现改错了
# 不要直接恢复！先告知用户
```

**错误做法（2026-05-01的教训）：**
```bash
# ❌ 直接用旧备份覆盖，丢失了用户的6路压力显示代码
cp App_lcd.c.bak App_lcd.c  # 这个备份是4月30日的旧版本！
```

**退档时的正确流程：**
```
我：发现改错了。当前有以下备份：
  - App_lcd.c.bak_20260501_140000 (修改前的版本，包含6路压力显示)
  - App_lcd.c.bak (4月30日的旧版本，没有6路压力显示)
  准备回到哪一份备份？

用户：回到 App_lcd.c.bak_20260501_140000

我：好的，执行恢复...
```
