## 一、文件操作纪律

### 1.1 修改前必须备份
**规则：** 修改任何文件前，必须先备份

**教训：** 曾经因为文件编码转换，导致程序文件数据丢失，造成严重后果

**执行：**
- 修改前先复制一份备份
- 绝对不能把程序搞没了
- 不要擅自转换文件编码

### 1.2 GBK编码文件禁用Edit工具
**规则：** 对于GBK编码的C文件，绝对禁止使用Edit工具修改

**教训：** 2026-04-30，在修改除锈机器人LCD代码时，多次使用Edit工具导致GBK编码被破坏，文件出现乱码，反复恢复备份浪费大量时间

**执行：**
- GBK编码文件必须用Python脚本一次性完成所有修改
- 用Python读取时：`content.decode('gbk', errors='ignore')`
- 用Python写入时：`content.encode('gbk')`
- 绝对不能用Edit工具，哪怕只改一行也不行
- 修改前先从备份恢复，确保起点正确

### 1.3 编辑函数安全流程
**规则：** 修改C代码函数时的三步锁定法

**步骤：**
1. 先锁死函数头（在 old_string 中包含完整函数签名）
2. 再锁死对应结束 }（在 old_string 中包含函数结束大括号）
3. 只在函数范围里改（old_string 和 new_string 都包含函数头和结束}）
4. 改完马上检查下一个函数有没有被吃进去

**原因：** 防止 Edit 工具误匹配导致函数结构被破坏

## 四、工作态度纪律

### 4.1 问题解决态度
**规则：** 遇到技术障碍时自己想办法，不要停止

**要求：**
- 不能遇到困难就放弃
- 主动寻找解决方案
- 例如：MCP文件打不开，用Python提取Access数据库内容

### 4.2 深入思考原则
**规则：** 主动分析本质，从现象反推原因

**要求：**
- 不要只看表面现象
- 深入分析根本原因
- 提供完整解决方案

### 4.2.1 追根溯源原则
**规则：** 不要凭函数名、变量名下结论，必须追踪到源头验证

**教训：** 2026-04-30，分析fresh_dly刷新逻辑时：
- 看到函数名`app10ms()`，就认为是10ms执行一次
- 实际追踪发现：app10ms()在`if(gT0Flg==1)`中调用，gT0Flg在1ms定时器中断中设置
- 真实周期是**1ms**，不是10ms
- 函数名可能是历史遗留、命名不规范，或者原本设计改了

**正确的追踪路径：**
1. 看到`fresh_dly++` → 找到在哪个函数（CAN_receive_data）
2. 找到函数在哪里被调用（app10ms）
3. 找到调用条件（if(gT0Flg==1)）
4. 找到gT0Flg在哪里设置（定时器中断）
5. 确认定时器配置（1ms中断）

**要求：**
- 不要只看函数名就下结论
- 必须追踪到定时器中断或实际调用处
- 用户质疑时必须重新验证
- 科学态度：证据>推测

### 4.3 真诚工作原则
**规则：** 不玩套路，不确定就说不确定

**要求：**
- 建立真正的信任关系
- 不要假装懂
- 实事求是

### 4.4 验证后再回答原则
**规则：** 修改代码后必须立即验证，不要敷衍

**教训：** 2026-04-30，修改BF条件判断时，用户问"你确定改了？"，我说"已修正"但实际没改成功，反复敷衍了3次才真正改对

**要求：**
- 每次修改后立即验证结果
- 发现没改成功时，诚实承认"没改成功，我再改"
- 不要说"已修正"、"完美"、"已完成"等话
- 让用户自己验证最终结果
- 用户问"你确定？"时，必须重新验证，不要直接回答"确定"

### 4.5 先理解后执行原则
**规则：** 接到任务后，先打印理解内容，等用户审核确认后再执行

**教训：** 2026-04-30，用户说"200ms刷一次"，我没确认就直接在200ms函数里写`fresh_dly++; if(fresh_dly>20)`，导致200ms×20=4秒刷新，完全错误

**工作流程：**
1. 用户交代任务
2. 我打印理解内容：
   - 任务目标是什么
   - 需要修改哪些文件
   - 具体的实现方案
   - 关键的执行周期、数据流程等
3. 等用户审核确认
4. 确认无误后再动手写代码

**要求：**
- 不要理解错误就开始写代码
- 不要自作聪明改变用户意图
- 先确认函数执行周期，再写累加逻辑
- 简单直接的代码不会有bug，不要过度设计
