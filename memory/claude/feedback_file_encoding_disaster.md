# 文件编码转换导致数据丢失 - 严重教训

## 事故概述
2026-04-26，在处理FSJ_666项目的App_lcd.c文件编码问题时，因错误的编码转换操作导致文件被完全清空（0字节），约1600行代码丢失，无法恢复。

## 为什么发生
用户反馈Keil打开文件显示乱码。我判断是编码问题，尝试用Python将文件从UTF-8转换为GBK。但是：
1. **没有创建备份**
2. 使用了 `errors='ignore'` 参数，导致无法编码的字符被丢弃
3. 文件原本可能就是GBK编码，用UTF-8读取产生乱码
4. 写入时文件被清空

## 强制规则（必须遵守）

### 核心原则：绝对不允许把整个程序搞没了

**用户的严厉警告（2026-04-26）：**
> "中文的解码确实比较麻烦，但是不能一刀切，把整个程序都搞没了。这个绝对不允许。"

这是最严重的错误：因为处理编码问题，导致整个源文件丢失。无论编码多么麻烦，**绝对不能让文件消失**。

### 1. 修改文件前必须备份
任何文件修改操作前，必须先创建带时间戳的备份：
```python
backup_path = file_path + '.backup_' + datetime.now().strftime('%Y%m%d_%H%M%S')
shutil.copy2(file_path, backup_path)
```

**这是底线，没有例外。**

### 2. 永远不要擅自转换编码
- 不要使用 `errors='ignore'`（会丢弃数据）
- 不要假设文件编码
- 必须先用chardet检测编码
- 优先使用Read/Edit工具（自动处理编码）

**中文编码确实麻烦，但不能因为麻烦就一刀切把文件搞没了。**

### 3. 遇到编码问题的正确做法
1. **停止操作**，不要尝试修复
2. **告知用户**问题原因
3. **建议用户**使用Notepad++等专业工具自己处理
4. **绝对不要**擅自进行编码转换

### 4. 操作后必须验证
```python
# 检查文件大小
if os.path.getsize(file_path) == 0:
    raise Exception("文件被清空！")
```

## 如何应用

**当用户反馈文件乱码时：**
1. 不要尝试转换编码
2. 告知用户：
   - 这是编码问题（GBK vs UTF-8）
   - 建议使用Notepad++查看和转换编码
   - 或者我可以提供修改建议，让用户自己操作
3. 如果必须操作：
   - 先创建备份
   - 用chardet检测编码
   - 操作后验证文件大小

**当需要修改包含中文的代码文件时：**
1. 优先使用Edit工具（自动处理编码）
2. 不要手动转换编码
3. 如果Edit失败，告知用户手动处理

## 记忆要点
- **修改前必须备份**
- **不要擅自转换编码**
- **使用工具自带的编码处理**
- **操作后必须验证**
- **遇到编码问题，让用户自己处理**

这是一次严重的失误，造成了无法挽回的数据丢失。必须永远记住这个教训。
