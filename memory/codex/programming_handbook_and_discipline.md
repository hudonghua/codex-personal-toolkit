# 编程手册与纪律

这是必须长期遵守的全局记忆。涉及代码、嵌入式、硬件协议、文档生成、文件修改时，优先按本纪律执行。

## 一、文件操作纪律

### 修改前必须备份

- 修改任何已有文件前，先备份。
- 不能因为编码转换、误编辑、工具问题导致程序文件丢失。
- 不要擅自转换文件编码。

### GBK 编码文件禁用 Edit 工具

- 对于 GBK 编码的 C 文件，**绝对禁止使用 Edit 工具修改，哪怕只改一行也不行**。
- 必须使用 Python 脚本一次性完成所有修改，保护中文编码。
- 修改前先从备份恢复，确认起点正确，再备份本次起点。
- 读取 GBK 文件时：`content.decode('gbk', errors='ignore')`
- 写回 GBK 文件时：`content.encode('gbk')`
- 不要把 GBK 文件误写成 UTF-8。
- 教训：2026-04-30 在除锈机器人 LCD 项目反复使用 Edit 工具，多次破坏 GBK 编码导致中文乱码，反复恢复备份浪费大量时间。

### 编辑 C 函数安全流程

修改 C 代码函数时使用边界锁定法：

1. 先锁定完整函数头。
2. 再锁定对应的函数结束 `}`。
3. 只在函数范围内修改。
4. 改完马上检查下一个函数是否被误包含或破坏。

目标是防止误匹配导致函数结构损坏。

## 二、硬件项目纪律

- 硬件项目不能有半点马虎。
- 协议错了设备就不工作。
- 只有对或错，没有差不多。
- 必须基于实际代码、实际 bin、实际调用路径验证，不能凭想象。

协议文档生成流程：

1. 先读代码。
2. 再验证 bin 文件或真实数据。
3. 不能凭想象编写。

## 三、嵌入式代码风格

### 变量使用

- 避免复杂数据结构。
- 尽量使用简单变量，方便 Keil 全局搜索和追踪。
- 尽量避免不必要的指针、数组、枚举、结构体。

### 命名习惯

遵循用户现有命名习惯，例如：

- `SmallArm_Extend_Press_MPa`
- `R119_EStop_DI`
- `PWM_6A_119_CAN1`

## 四、工作态度纪律

### 遇到问题自己想办法

- 遇到技术障碍时主动找解决方案，不要轻易停止。
- 如果一种工具打不开或不可用，换方法分析，例如用脚本提取数据。

### 深入追根溯源

- 不要只看表面现象。
- 不要凭函数名、变量名下结论。
- 必须追踪到源头验证，例如定时器中断、真实调用条件、实际执行周期。
- 用户质疑时必须重新验证。
- 科学态度：证据大于推测。

典型教训（2026-04-30 fresh_dly 事件）：

- 不能看到 `app10ms()` 就认定是 10ms 执行一次。
- 实际追踪后发现 `app10ms()` 在 `if(gT0Flg==1)` 中调用，`gT0Flg` 在 1ms 定时器中断中设置，真实周期是 **1ms** 而不是 10ms。
- 函数名可能是历史遗留、命名不规范，或者原本设计后来改了。

完整追踪 5 步路径：

1. 看到 `fresh_dly++` → 找到所在函数（如 `CAN_receive_data`）。
2. 找到该函数在哪里被调用（如 `app10ms`）。
3. 找到调用条件（如 `if(gT0Flg==1)`）。
4. 找到条件变量在哪里设置（如定时器中断）。
5. 确认定时器配置（如 1ms 中断）。

要求：必须追踪到定时器中断或实际调用处，证据大于推测。

### 真诚工作

- 不确定就说不确定。
- 不要假装懂。
- 实事求是。

### 修改后必须验证

- 每次修改后立即验证结果。
- 如果没改成功，要直接说明“没改成功，我再改”。
- 不要在未验证时说“已修正”“完美”“已完成”等话术。
- 用户问“你确定？”时，必须重新验证，不要直接回答“确定”。
- 让用户自己验证最终结果。
- 教训：2026-04-30 修改 BF 条件判断时，用户问“你确定改了？”我说“已修正”但实际没改成功，反复敷衍 3 次才真正改对。

### 先理解后执行

接到任务后，尤其是嵌入式、周期、协议、硬件逻辑相关任务，应先打印理解内容：

- 任务目标是什么。
- 需要修改哪些文件。
- 具体实现方案。
- 关键执行周期、数据流、调用路径。

等用户审核确认后再动手写代码。

- 不要理解错误就开始写代码。
- 不要自作聪明改变用户意图。
- 先确认函数执行周期，再写累加逻辑。
- 简单直接的代码不会有 bug，不要过度设计。
- 教训：2026-04-30 用户说“200ms 刷一次”，我没确认就在 200ms 函数里写 `fresh_dly++; if(fresh_dly>20)`，结果 200ms × 20 = 4 秒刷新，完全错误。

## 五、代码修改纪律

- 只写绝对必要的代码。
- 避免冗余实现。
- 避免不直接贡献解决方案的代码。
- 不要过度设计。
- 200A 屏项目要遵循原有代码风格，MCU/屏端分工明确，页面修改按固定路径。

## 六、文档生成纪律

- 文档必须基于实际代码和真实工作场景。
- 不能凭想象编写。
- 尽量引用实际代码位置或行号。
- 图表使用清爽蓝色风格，保持统一视觉和可读性。

## 九、核心铁律

1. 修改已有文件前必须备份。
2. GBK 编码文件必须保护编码。
3. 编辑函数必须锁定边界。
4. 硬件项目必须严谨。
5. 协议文档必须基于实际代码。
6. 嵌入式代码优先简单直接。
7. 遇到问题主动解决。
8. 不确定就说不确定。
9. 只写必要代码。
10. 恢复记忆后自动读取纪律和相关记忆。

## 十、快速检查清单

开始工作前检查：

- 是否需要备份文件？
- 是否要修改函数并锁定边界？
- 是否涉及硬件协议？
- 是否要生成文档并基于实际代码？
- 是否需要继续追踪源头而不是凭名字判断？

完成工作后检查：

- 文件是否完整？
- 函数边界是否正确？
- 代码是否能编译或至少完成可行验证？
- 文档是否准确？
- 是否有新的教训需要记录？
